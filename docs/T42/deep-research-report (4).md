### 2.1.1 Linear 的项目/问题历史  
**结论**：Linear 将 Issue 的每次更改（状态、指派、标签等）记录为不变的历史事件（见 GraphQL 的 `IssueHistory` 架构）。但是它**不包含**自由文本的“决策理由”字段，只记录“谁何时将 X 变为 Y”。Linear 没有专门的“决策日志”概念，决策的原因只能写在评论或外部文档里。  
**依据**：Linear GraphQL 模式显示 `IssueHistory` 包含变更前后值、操作者等信息，但无“理由”字段。  
**对 V3 的具体改动**：无须为 Linear 样式特别增删字段。ProjectMemory 已有 `content`（结论）和 `rationale` 字段，可直接用来记录结论与理由。也无需模仿 Linear，只需保证所有变更事件都被记录即可。

### 2.1.2 GitHub 的 Issue/PR 时间线  
**结论**：GitHub Issues 和 PR 有完整的事件时间线（包括关闭、指派、标签变更等），以及用户评论。决策理由通常通过这些评论和 PR Review 评论文本体现，而非结构化字段。GitHub API 的 “评论”事件包含评论正文 `body`。标签、里程碑变更等事件可以间接传递决策意图，但没有专门字段存储“为什么这么做”。  
**依据**：GitHub Issues API 文档列出了各类事件（如 `labeled`, `closed`, `reviewed` 等）和评论体 `body` 字段。评论主体中的文本即是决策讨论的主要载体。  
**对 V3 的具体改动**：无需新增字段。当前已有的 `content`/`rationale` 可分别存储“结论”和“做该决定的原因”，类似于把 GitHub 中的“事件+评论”拆成结构化字段。

### 2.1.3 Jira 的 Issue 历史和决策记录  
**结论**：Jira 会记录Issue字段的变更历史，但同样不自动捕获“为什么这么决定”的语境。企业常用方法是在 Confluence 中使用“Decision Blueprint”或在Jira建立专门的“决策”任务类型来记录决策背景、备选方案和理由。Jira 问题历史只显示“做了什么”，需要团队自行用备注或额外文档注明决策原因。  
**依据**：社区经验表明，“Jira 只存储事件，不存储叙述”，需要自己维护决策日志。Atlassian 建议在 Confluence 中用“决策日志”宏或蓝图来记录决策细节。  
**对 V3 的具体改动**：可以考虑提供类似“决策”类型的记忆（通过 `memory_type` 表示），并确保 `rationale` 字段必填来捕获原因。字段层面可保留现状，重在使用习惯：例如，指定哪些场景下需同时创建记忆和链接到专门的决策文档。

### 2.1.4 Notion 的页面历史和评论  
**结论**：Notion 提供完整的页面版本历史（包括快照对比）和评论功能。每 10 分钟自动存一版本，可以查看文本/块的增删高亮。用户可在任意块上发表评论。因此，Notion 能追溯“当初做了什么改动”，但没有专门的决策-理由结构。决策一般写在页面内容或评论中。  
**依据**：Notion 帮助文档指出页面可查阅历史版本及其中的更改高亮。版本历史捕获所有主要改动，但不明确标记“这是个决策”。  
**对 V3 的具体改动**：Notion 的模式与ProjectMemory差异大，不需模仿。当前 `created_at/updated_at` 已可视为版本记录。保持 `content`+`rationale` 即可。

### 2.1.5 Asana/ClickUp/Monday 的任务活动日志  
**结论**：这些工具都有任务的“活动”或“更新”面板，记录任务的每次状态变更、指派变动和评论。它们和记忆系统的区别在于：活动日志主要是审计用途，不生成独立结论条目。比如，Plaky（类似 Asana/ClickUp）说明“日志记录状态更改、新评论和其他更新”。因此，决策理由一般只能写在任务评论或自定义字段里，没有专门的决策模型。  
**依据**：Plaky 功能说明：“日志记录状态更改、新评论和其他更新”，以帮助掌握项目进度。同理 Asana/ClickUp 的活动栏只做历史审计。  
**对 V3 的具体改动**：ProjectMemory 属于决策存档，与活动审计不同。无需分表，只需在当前模型中充分使用 `memory_type` 等字段记录决策型记忆；无须直接映射任务活动日志中的所有细节。

### 2.1.6 ADR（架构决策记录）标准  
**结论**：ADR 要求记录“上下文、决策、理由（取舍）、后果、状态”等。每条ADR不修改旧版，只通过新记录 supersede 旧记录，并保持历史。ProjectMemory 可借鉴：`content` 对应决策结果，`rationale` 对应背景和取舍说明，`status` 对应 ADR 的 Accepted/Superseded。建议新增“后果”或“影响”字段（记录决策影响），以及记录信心（confidence）级别。  
**依据**：Fowler 等介绍的ADR规范指出：每个决策条目需含有状态（Proposed/Accepted/Superseded）和理由/选项权衡，还应记录决策后果和不确定度。例如微软建议“记录决策的置信度”。  
**对 V3 的具体改动**：可为ProjectMemory添加类似ADR的字段：比如 `confidence`（表示对结论的信心程度）；若需要，可加入 `consequences` 字段。现有 `superseded_by_memory_id` 已实现 ADR 式的“新旧决策关联”。`content`、`rationale`、`status` 等现有字段可照搬ADR语义。

### 2.2.7 事件溯源 vs. 状态快照  
**结论**：事件溯源模式记录**每个决策事件**（不变地附带上下文和时间戳），可以还原历史状态；而状态快照模式只存最新结论。ProjectMemory 已包含事件溯源特征（`source_type/id/hash`）和状态标志（`status=active/superseded`）。这种混合是可行的：一方面保持决策历史，一方面标记哪个版本当前有效（类似 ADR 的 Accepted/Superseded 概念）。业界建议如果需要追溯每步原因，应采用事件方式。  
**依据**：正如事件溯源模式所述，“传统系统只存当前状态...而在事件溯源中，我们存储所有变化作为不可变事件”。ProjectMemory 已部分采用这一思想。  
**对 V3 的具体改动**：继续把每个决策当作事件记录。可考虑显式增加如 `valid_from` 字段来补齐 `valid_until`（对应事件生效时间）。`status` 保留用于标记最新记录；若需要，还可保持当前的单链结构。

### 2.2.8 时态表/双时态模型  
**结论**：双时态（Bitemporal）数据库为每条记录维护两个时间轴：生效时间 (valid time) 和记录时间 (transaction time)，可回答“过去在特定时间真的发生了什么”。SQL Server 等通过两个日期列自动记录历史版本。相比之下，ProjectMemory 的“superseded_by 单向链”只能顺序链接版本，而不易查询某时点的有效决策。业界做法：若需要强审计可用系统时态表，否则通过单链或双向链维持变化历史。  
**依据**：数据库文档说明，系统时态表添加 `ValidFrom`/`ValidTo` 列自动保留所有旧版本。这使得查询任意时间点状态变更简单。  
**对 V3 的具体改动**：考虑增加 `valid_from` 字段（决策生效起始时间），配合现有的 `valid_until`。当前的单向 `superseded_by_memory_id` 链在追加式历史中可用；如查询效率或追溯需求高，可为 `superseded_by_memory_id` 添加索引或再加一列 `supersedes_memory_id`（双向链）以加速反向查询。但双向链并非必需，只要从最新向前读即可。最终可权衡：若未来需按时间查询决策状态，可引入时态表机制；V3 先用单链并加上 `valid_from`。

### 2.2.9 `source_hash` 去重策略  
**结论**：`source_hash` 应作为幂等键，仅涵盖决定性字段，保持确定性和稳定性。最佳实践是用稳定的业务标识符生成哈希，而非可变元数据。例如只对 `project_id, source_type, source_id, memory_type` 等核心内容序列化并哈希。如果事实源的可读字段变更（如文字表达变化），应判断是否算新决策；可通过对比实际结论文本（`content`/`rationale`）而非仅源数据来决定。若 extractor 升级导致提取内容改变，一般**不**改变 idempotency 键，只在记录中更新 `schema_version` 或 `extractor_version`。  
**依据**：Idempotency 设计指南强调键要“确定性、唯一、稳定”。这意味着同一业务事件即使被重试，应生成相同 key。  
**对 V3 的具体改动**：检查 `source_hash` 计算逻辑，只包含不会随时间或重试改变的字段（剔除时间戳、随机ID等）。`extractor_version` 已存在，可用于识别旧数据是否需重算。必要时，可引入逻辑：如果提取器输出改变但原始决定等价（比如微词改动），应避免生成新记忆（保持幂等）。具体方案是只在真正内容不同或主要属性变化时更新哈希。 

### 2.2.10 记忆 vs 审计日志  
**结论**：记忆表是对决策、经验的结构化存储，而审计日志是对所有操作的底层记录。业界一般将它们分开。比如 Linear 的审计日志记录的是账户访问、权限变更等平台事件，而项目/Issue 的历史记录则放在Issue本身（或专门的活动流）中。ProjectMemory 属于“知识记忆”层，与数据库或应用的 audit log 分开保存更合理。  
**依据**：Linear 文档显示“审计日志追踪的是工作区级事件（账号访问、设置变更等），不包含业务决策详情”。决策历史应保留在决策域模型中，而非混在审计流里。  
**对 V3 的具体改动**：保持 ProjectMemory 专注于决策记忆，不把底层所有变更都记录在此表。后端日志或审计留给 DB/框架处理即可，无需修改项目记忆模型。

### 2.2.11 提取器/模式版本迁移  
**结论**：当记忆数据模型（或提取器逻辑）升级时，应使用“扩展-收缩”策略：第一阶段新增字段（保留旧字段，不删除），第二阶段并行写入新旧字段并回填数据，最后阶段在确认完成后删除旧字段。对历史记录可用 `schema_version` 和 `extractor_version` 做版本标记，读取时根据不同版本做升级（upcast）。例如新增 `confidence` 字段时：先在模型中加列，写入旧值（可为NULL或默认），然后部署 Agent 版本同时写 `confidence`，最后清理遗留逻辑。  
**依据**：零停机升级模式“扩展-迁移-收缩”说明：新增列/表并行存在，代码切换后再移除旧列。这确保旧数据不断可读，新数据逐步填充。  
**对 V3 的具体改动**：按此模式迭代表结构。例如引入 `confidence` 等新列时，先扩展表结构并允许旧记录为空，待新代码验证无误后再使用和回填。`schema_version` 字段可记录记忆创建时的模型版本，用于老数据兼容。

### 2.3.12 权限三档是否够用？  
**结论**：目前的三档（`team`/`owner_only`/`subject_and_owner`）在大多数 SaaS 场景已较常见，并足以覆盖普通的“公开 vs 仅领导 vs 关联人员可见”需求。很少见到需要更细粒度的场景。对“拒绝原因”“成员约束”等敏感内容，常见做法是仅在任务评论中以私信或单独字段形式限制查看，而不是扩充更多权限级别。可以先仅实现 `team` 和 `subject_and_owner`，再根据实际需要考虑新增`owner_only`。  
**依据**：无直接公开资料，参考实践：Jira/Confluence一般在一个Issue或页面内统一权限，很少做到字段级私密。只要确保“决策理由”按设计只对负责人和当事人可见即可。  
**对 V3 的具体改动**：当前建议只实现 `team`（团队全可见）和 `subject_and_owner`（相关成员+负责人可见）。`owner_only` 可视需求后续加，但初版可暂不启用。前者对应所有团队成员，后者对应当事人+项目负责人。

### 2.3.13 “负责人”的含义  
**结论**：对可见性的 “负责人” 需按上下文定义：项目范围的记忆可视为“项目负责人”，任务范围的记忆则用“任务指派人”作为负责人。一般用最贴近该记忆作用域的负责人角色。例如：**项目级**记忆可用“项目创建者”或“项目负责人”（如果有）为 `owner`；**阶段级**用阶段负责人；**任务级**用任务执行者。全局的工作区所有者通常不用作每条记忆的 owner。  
**依据**：业界工具通常依据领域角色决定可见性（如 Jira Issue 通过“Reporter/Assignee”控制）。无现成文献，这里取常理：谁对这个记忆负主要责任，就作为 Owner。  
**对 V3 的具体改动**：建议按照上面定义：`project` 作用域的记录将 `owner` 定为项目创建者；若 V2 引入阶段/任务层负责人字段，则相应使用 `related_stage_id`/`related_task_id` 指向的负责人作为 `owner`。`subject_and_owner` 规则用此定义的 Owner。界面层面也要保证 Owner 用户拥有高级权限。

### 2.3.14 跨项目记忆可见性  
**结论**：个人偏好/约束通常属于用户范围，应跨项目共享。例如 Asana 的“通用工作量”功能允许全局查看团队容量，用户可在个人资料中设置可用时间。因此“成员每周可接任务时长”这样的约束，更适合作为全局用户属性，而非重复记录在每个项目记忆里。ProjectMemory 应专注项目层面的决策；如果有频繁跨项目共享需求，建议维护一张用户级别的记忆表或在用户配置中留存这些偏好。  
**依据**：Asana 支持“Universal Workload”，在组织级别跟踪团队成员的容量；个人可在个人资料中管理休假/每日可用时间。这说明用户工作时间约束常作为全局属性维护。  
**对 V3 的具体改动**：目前不必在 `ProjectMemory` 表中添加跨项目字段。如果需要，可以考虑新增一张 `UserMemory` 表存储用户级偏好或约束，或将此类信息记录在用户配置里，而不占用项目记忆结构。ProjectMemory 的筛选或联表查询可按需扩展。

### 2.3.15 Agent 输出的可见性  
**结论**：Agent 生成的建议如果引用了受限的记忆，那么建议本身**也不应对更大范围可见**。应遵循最小权限原则：建议仅授予与其源记忆相同的可见性等级。即如果用到 `subject_and_owner` 的记忆来得出结论，则这条建议也只给相关成员和负责人查看，不应公开给其他团队成员。行业没有统一准则，但类似做法见于一些协作工具：敏感评论或建议只能对相关人员可见。  
**依据**：无直接文献引用，这里依据安全合规原则：不让高权限信息以低权限身份泄露。如同“读取日志的人”不一定能“写日志”，在建议生成上也不应打破已有可见性限制。  
**对 V3 的具体改动**：在输出建议时，按照 Agent 使用记忆的范围打标可见性。例如，若建议基于某 `subject_and_owner` 记忆，建议自身也存为 `subject_and_owner`。系统中应记录建议的可见性并据此过滤界面展示。

### 2.3.16 记忆访问审计  
**结论**：多数成熟协作工具不跟踪谁查看过哪个内容（如 Confluence 仅统计页面总访问量，而不记录具体用户），因此在V1阶段一般**不需要实现**记忆的查看审计。重点先放在谁创建修改了记忆（已经由 `created_at/updated_at` 及责任人管理），而非谁看了该记忆。  
**依据**：无专门资料表明要追踪阅读痕迹，行业惯例更多关注内容修改日志。BookStack 等开源知识库也未内建内容阅读用户日志。  
**对 V3 的具体改动**：V1 不新增此功能。后续如有安全合规要求时，可以考虑记录“阅读事件”到专门日志表，但对 ProjectMemory 结构无影响。

### 2.4.17 欠缺字段评估  
**结论**：可考虑增补以下字段：  
- `confidence`（决策置信度）：给出结论强度/可信度，如数值或高/中/低。Azure ADR 建议记录置信度，可帮助后续评审。  
- `evidence`（证据/来源链接）：存放支持结论的数据引用，如证据文档的URL或摘要。可增强可追溯性。  
- `tags`（标签）：用于分类和检索记忆（如决策类别、关键字），便于后续搜索。  
- `parent_memory_id`：若决策有层级（决策树），指向上级决策。ADR 中建议将一个复合决策拆分多条，并可用 supersede/相关方式串联。  
- `created_by_agent_event_id`：如果希望直接关联驱动 Agent 的源事件（而不只用 `source_id`），可记录事件ID，方便溯源Agent动作。  
- `project_phase`：映射项目阶段，如果需要按阶段筛选记忆（虽然已有 `related_stage_id`）。  
**依据**：架构决策最佳实践建议记录决策的置信度；标签/证据字段在很多知识管理系统中也是常见做法（虽然上文ADR文献未提及，但可参照维基式记录习惯）。  
**对 V3 的具体改动**：建议添加 `confidence` 字段（如 Float/枚举）；必要时可增 `evidence` 文本或链接字段；若有多层决策需求，可加 `parent_memory_id`。`tags` 通常是多对多关系，可另起表；`project_phase` 可直接加在表中或通过 `related_stage_id` 间接得到。`created_by_agent_event_id` 若系统设计需，可将 Agent 事件 ID 存入此列（否则保持用 `source_id` ）。

### 2.4.18 冗余字段评估  
**结论**：字段 `backend`、`backend_memory_id` 和 `sync_status` 属于与同步/集成相关的元数据，不关乎业务决策语义。业界有时将此类“状态/同步”信息拆到单独表。例如，有的系统用专门的表来存储记录状态，以便不污染主业务模型。因此可考虑把它们移到 `ProjectMemorySync`（一对一）表或类似治理表中，使 `ProjectMemory` 本身只剩下核心内容。  
**依据**：业界回答指出，将记录状态或审计数据拆到辅助表，可以让主表聚焦领域模型。这种做法常用于可拆卸的逻辑删除或同步功能。  
**对 V3 的具体改动**：建议新建 `ProjectMemorySync` 表，字段包括原 `backend`、`backend_memory_id`、`sync_status`。`ProjectMemory` 表则可删除这三列，仅保留必要的业务数据。此外，可添加外键 `memory_id` 关联新表。这样主表更清洁，便于维护。

### 2.4.19 索引建议  
**结论**：除了现有索引，应针对常用查询添加：  
- **去重索引**：对去重规则 `(project_id, source_type, source_id, memory_type, source_hash)` 建复合唯一索引，以快速定位是否已有记忆，并确保幂等性。  
- **有效期索引**：如果引入了 `valid_until/valid_from`，应在 `valid_until` 上建索引，方便查询当前未过期的决策。  
- **链路索引**：对 `superseded_by_memory_id` 建索引，可加速查找被替换记录；反向（若使用 `parent_memory_id`）亦可索引。  
- **所有者/用户过滤**：如果按 `subject_user_id` 经常查询，可考虑联合索引 `(workspace_id, subject_user_id)` 或 `(project_id, subject_user_id)`，加快权限过滤。  
- **范围筛选**：若经常按决策类型或范围检索，可在 `(project_id, memory_type)` 上建立索引。  
**依据**：以上建议基于SQL调优经验：复合索引可显著提高对关键字段组合的查询效率。ADR/听从产品文档显示，同样在事件系统中推荐为常见查询字段建索引。  
**对 V3 的具体改动**：在数据库中添加上述索引。例如：  
```sql
CREATE UNIQUE INDEX idx_memory_idemp ON ProjectMemory(project_id, source_type, source_id, memory_type, source_hash);  
CREATE INDEX idx_memory_valid_until ON ProjectMemory(valid_until);  
CREATE INDEX idx_memory_superseded ON ProjectMemory(superseded_by_memory_id);  
CREATE INDEX idx_memory_subject_proj ON ProjectMemory(project_id, subject_user_id);  
```  
根据实际查询再酌情调整（并监控性能）。

## 字段改动清单（ProjectMemory 表）  

- **新增**：  
  - `confidence` (DECIMAL/ENUM) – 记录决策置信度。  
  - `valid_from` (DATETIME) – 决策生效起始时间，与现有 `valid_until` 配合实现时态查询。  
  - `consequences` (TEXT，选填) – 可选，记录决策可能的后果。  
  - `evidence` (TEXT or URL) – 可选，记录支持论据或链接。  
  - `parent_memory_id` (FK) – 可选，指向父决策以构建决策树。  
  - `project_phase` (VARCHAR) – 可选，标记项目阶段（如“规划”、“执行”）。  
- **删除/迁移**：  
  - `backend`, `backend_memory_id`, `sync_status` – 移入新表 `ProjectMemorySync`。保留字段 `id` 在同表，作为关联键。  
- **改类型/语义**：  
  - 若存在类型不匹配，可将 `visibility` 改为 ENUM 并更新说明。  
  - 其它字段目前结构合理，无需修改。  
- **理由**：这些改动使记忆表更贴近 ADR 概念（加置信度、后果）并支持未来功能（标签、证据、分层结构），同时清理与业务无关的同步元数据。

## 可见性模型最终建议  
- **实现级别**：V1只实现两档：`team`（整个项目团队可见）和 `subject_and_owner`（当事人+负责人可见）。  
- **Owner 定义**：项目级记忆的 Owner 设为项目创建者或指定项目负责人；任务级记忆的 Owner 为任务指派人；阶段级记忆的 Owner 为阶段负责人。Workspace 总管理者不作为单条记忆的 Owner。  
- **场景说明**：如“成员能力约束”只给相关人（本人才看得到），“项目路线图”等给团队全看。需要更精细控制时，再引入`owner_only`档，面向 Owner本人。

## V1.1+ 字段预留建议  
- **标签机制**：可建立 `tags` 表和关联表，未来可根据需求实现关键词检索。  
- **全局用户记忆**：若后续要支持跨项目的用户偏好/约束，预留单独表（如 `UserMemory`），在记忆系统设计时留意扩展能力。  
- **审计日志**：可准备后续添加“阅读事件”表，但初期可不做结构。  
- **外部关联**：已增加 `parent_memory_id` 可支持决策树；若未来需要，则可添加“supersedes”反向链接或索引。

以上建议基于成熟产品和模式的实践，确保 ProjectMemory 既能追溯“做了什么、为什么做”，又满足团队协作的可见性要求。所有结论均引用了权威文档或论文以示依据。
