## Part C：关键判断题

**13. 纯 SQLite FTS5+嵌入自研 vs Mem0 的权衡**：对于 ProjectFlow（每项目记忆条目仅百～千条，中等召回要求），自研方案**可行**且运维最简，不依赖外部框架或服务。SQLite FTS5 做关键词检索，配合事先用句向量计算的语义匹配+简易重排序，可满足基础需求，省却 Mem0 这类框架的学习与运维成本。缺点是召回质量和丰富度可能逊色于专门记忆系统（如缺乏稠密检索算法优化、智能去重等）；同时，开发和维护匹配策略需要投入工程量。若团队不介意稍低的检索性能，自研可在预算和控制风险上获益；反之，Mem0 提供了开箱即用的结构化记忆算法和成熟接口，可节省调优时间。**推荐**：若资源紧张，倾向自研；若追求最好召回质量和未来可扩展性，使用 Mem0OSS。

**14. V2 升级路径 (引入 Graphiti)**：如果 V1 用 Mem0 OSS 存储事件，在 V2 引入 Graphiti 时，需要将 ProjectFlow 事件/记录映射为图实体和关系。前置设计包括：确保 `ProjectMemory` 表或相关事件表中包含清晰的实体标识、时间戳和属性字段，方便提取节点（如用户、任务等）和边（如“执行”、“创建”关联）。可采用事件流模型：每条记录当作图的事务性条目。架构上需保留原始事件以便构建知识图；在设计表时最好标记事件类型和参与实体、时间线，以利后续自动转换。若初期设计得当，迁移成本可控；否则需编写批量导出 Mem0 中数据并通过 Graphiti API 逐条创建节点/边。

**15. “接管 runtime” 红线**：根据 T41 要求，不得使用那类必须托管 Agent 循环的框架。**Letta** 是典型的“接管”例子——它提供了自己的 Agent 循环和生态，要求将对话交给 Letta 平台。**Cognee、Zep、LangMem/LangGraph** 均只是存储/检索库，没有控制 Agent 流程，因此**不会**强制交出 runtime。简言之：**Le

tta/Mapverse 等 Agent 平台会接管循环，非选；Cognee/Graphiti/Weaviate/Chroma 等作为库/服务使用，可符合要求**。

**16. 依赖雪球**：  
- **Mem0 OSS**：Python 库，依赖一个向量数据库（默认 Qdrant 客户端），可选 Redis、Weaviate、Chroma、Milvus 等。对现有 FastAPI+SQLite 无直接冲突，只是增加可选组件。  
- **Graphiti**：Python 库，依赖 Neo4j 5.x（或 FalkorDB/Kuzu 替代）；需部署图数据库。  
- **Zep Cloud/OSS**：Cloud 方案无需本地依赖，OSS SDK 仅需 Python 客户端，不需数据库。  
- **Cognee**：Python 库，依赖 SQLite（aiosqlite）、LanceDB（向量），Pydantic 等；与 FastAPI/SQLModel 完全兼容。  
- **Letta**：主打 Node.js/TypeScript，需要 Node 环境、FastAPI（如 V1 支持）、数据库（SQLite、Postgres、Redis 可选）。会引入跨语言技术栈。  
- **LangMem/LangGraph**：依赖 LangChain 库及其后端（Python），LangGraph 存储需要 Postgres（异步驱动、pgvector）。额外引入 LangChain 依赖。  
- **Mem0 平台**：无需额外依赖（调用云 API）。  
- **sqlite-vec/vss**：纯 SQLite 扩展，无其他依赖。  
- **pgvector**：需要切到 PostgreSQL 数据库（部署或迁移）。  
- **Chroma**：Python 库，可选择内存运行或持久运行，不需外部数据库。  
- **Milvus**：独立服务（Go），需 Docker/K8s 环境，可能需 GPU 支持（可选）。  
- **Weaviate**：独立服务（Go），通常以 Docker/K8s 部署。  
- **其他**：如 Marqo 等已弃用，A-MEM 等研究性项目无生产级依赖。

总的来说，核心后端栈（Python 3.11、FastAPI、SQLModel）与大部分候选兼容。唯一需要注意的强制变动是**数据库类型**：若选 sqlite-vec 或 Mem0 OSS V1，无需改动；若将来迁移到 Postgres，可使用 pgvector；Graphiti、Milvus、Weaviate 等需要额外服务，Letta 则引入 Node.js。

## Part D：最终推荐

- **V1 检索层选型**：基于权衡，**推荐采用 Mem0 OSS**。理由：Mem0 提供成熟的记忆存储和检索算法，与已有 Python/FastAPI 环境兼容（见官方文档）。同时建议并行评估**自研 SQLite+向量**方案作为备选：小规模下快速迭代，且无需引入外部依赖。若团队优先保证高召回和可扩展性，用 Mem0；若追求简单及本地完全控制，可选用 SQLite 向量扩展（如 sqlite-vec）或纯 FTS5 实现。  
- **V2 升级路径排序**：优先考虑引入**Graphiti 时序图**和**Cognee 知识图**作为增强选项。具体可行顺序：① 若项目需要复杂时序/关系查询，可在 Mem0 上游增加 Graphiti 事件记录（该方案需项目设计事件流）；② 若需要更高级的知识管理，可采用 Cognee 作为持久化知识库；③ 兼顾方案：在 Postgres 环境下引入 pgvector 替代 SQLite 向量，配合 LangGraph/LangMem 思路辅助 Agent。  
- **选择 Mem0 OSS 的前置条件**：确保 `ProjectMemory` 表设计支持可量化的「添加-检索」流程（包括清晰的实体标签和作用域），以便 Mem0 能准确关联。需要配置并启动支持的向量数据库（如 Qdrant），或选择 Mem0 自带的向量存储。保持 FastAPI 作为事实写入惟一入口，Mem0 调用应仅在“草稿”到“建议”生成等环节中出现。详细参考 Mem0 平台文档。此外，若项目合规要求，本地部署以避免云服务数据出境。

**总之**：V1 采用 Mem0 OSS（或自研 SQLite 向量扩展），V2 考虑引入 Graphiti/Cognee 等知识图方案。若选 Mem0 OSS，上述设计前提务必落地，确保数据结构化存储并易于未来迁移。文中各项结论均基于官方文档和 GitHub 资料。
