### 2.1.1 LoCoMo  
**结论**：LoCoMo 是 Snap Research 2024 年提出的「超长对话记忆评测基准」，旨在评估对话型 AI 在极长对话中的记忆能力。数据集包含 50 条对话，每条对话约 300 轮、9200 词左右，并在多轮会话中引入人物情景、事件时间线和图像共享。评测任务包括多轮问答（QA）、事件图总结和多模态对话生成。指标方面，对问答使用回答精度（F1）和检索召回率（Recall@k）来衡量；事件总结使用事实精度/召回（FactScore）及 ROUGE 等指标。  
对于 ProjectFlow 这样的“项目决策”场景，LoCoMo 主要考察人物对话记忆，不完全吻合。不过其设计思路可借鉴：可以将项目日志拟作多轮会话，设计类似的 QA 任务（例如询问“为什么做出某个决策”）或总结任务（如总结项目进度事件），来检测模型能否回忆相关记忆。LoCoMo 的指标定义（回答准确率、检索召回率、事实一致性度量等）也可参考。  

**依据**：Maharana 等人《Evaluating Very Long-Term Conversational Memory of LLM Agents》；Mem0 文档中的 LoCoMo 说明。  
**V1/V1.1 归属**：建议将 LoCoMo 的评测任务和指标作为 V1.1 及以后的改进目标（直接采用需调整到项目语境）。  
**对 V3 的具体改动**：可参考其问答和总结任务框架，在评测 harness 中加入类似形式的项目问答/事件总结，用于检验记忆检索效果（暂不直接纳入 V3 设计）。  

### 2.1.2 LongMemEval  
**结论**：LongMemEval 是 2024/25 年提出的聊天助手长期记忆基准，共有 500 个高质量的提问，覆盖五大记忆能力：信息提取、多会话推理、时间推理、知识更新和回避回答。每个问题都依赖于多轮用户–AI 对话历史，标准评测设置下对话历史长度分别约为 115K 词和 1.5M 词。评价指标包括检索准确率（Recall@K、NDCG）和最终回答正确率。商业大模型在 LongMemEval 上表现出大幅性能下降，可见其难度之高。  
与 ProjectFlow 场景相比，LongMemEval 强调“多会话聊天记忆”，关注在海量对话历史中跨回合推理（如关联信息、时间线问题）。ProjectFlow 的“多轮项目推进决策”虽然也是跨多轮上下文，但更结构化（任务/计划、成员分配等）。部分能力如多会话推理和时间推理是共通的：可以设计测试问答题目如“团队在最初为何选择做 X”、“任务 A 与 B 的时间先后如何影响计划”，来检验模型跨阶段记忆。但 LongMemEval 包含的“回避无解”、“动态信息更新”等能力在项目场景可能不完全适用。总体而言，可以借鉴其大规模、多轮历史设定，但需将问题重构为项目业务问题。  

**依据**：Wu 等人《LongMemEval: Benchmarking Chat Assistants on Long-Term Interactive Memory》。  
**V1/V1.1 归属**：LongMemEval 的分析和题型适合在 V1.1 中考虑（V1 可参考其思路进行项目问答构造，当前无必要完全复制）。  
**对 V3 的具体改动**：若采用，可在评测套件中添加类似多轮项目问答题目，用于衡量跨阶段记忆召回和问答准确度。  

### 2.1.3 BEAM  
**结论**：BEAM（Benchmark for Evaluating Agentic Memory）是 Mem0 提供的记忆检索评测框架，专注于大规模上下文（1M、10M 词量级）的记忆能力测试。它涵盖 10 类任务（如偏好/指令追踪、信息提取、知识更新、多轮推理、时间推理、事件排序、回避无解、矛盾解决等）。BEAM 测试在 1M 和 10M 级别上下文下模型的检索准确率（通过 pass-rate/平均得分衡量，各类别精度列于 Mem0 文档表中）。该框架开源（可通过 Mem0 memory-benchmarks 仓库使用），但需要预先准备大规模会话日志用于测试。  
因 BEAM 规模巨大（最大 10M 词上下文），直接运行成本很高。对于 ProjectFlow，可从 BEAM 中借鉴“不同维度记忆任务”和多上下文测试的思路。V1 阶段可以选取 BEAM 中与项目相关的子任务（如信息提取、时间推理、多会话推理）构造小规模测试；若资源充足，V1.1 可尝试运行 Mem0 提供的 BEAM 测试套件以全面评估。  

**依据**：Mem0 官方文档“Memory Evaluation” 中 BEAM 部分。  
**V1/V1.1 归属**：BEAM 属于较重的长上下文评测，可作为 V1.1 的扩展目标。V1 阶段建议先落地简单检索任务。  
**对 V3 的具体改动**：如启用，可在检索流程中增加大规模上下文测试，或调整 top_k、检索策略以应对更大记忆库。  

### 2.1.4 ProjectFlow 自建评测 Harness  
**结论**：为验证记忆检索流程，可设计一个最小化的“假 LLM”评测套件：预置一个固定的项目示例（包含若干任务、成员、里程碑等），人工创建对应的记忆条目（例如决策缘由、风险说明、分工逻辑等）。然后设计一系列检索场景：基于当前对话上下文调用检索流程，并检查返回的记忆是否符合预期。使用断言来校验：相关记忆应被检索到（至少一个相关条目出现在 top_k），无关记忆不应出现。指标可定义为**检索准确率**（被检索出的相关记忆数 / 总相关记忆数）、**误召率**（检索结果中无关记忆占比）等。例如，在场景“澄清决策原因”中，断言包含当初记录的该决策动机的记忆，而不包含其他无关任务记忆。可以借鉴 LoCoMo/LongMemEval 的思路，将特定问题（“我们为何做 X?”、“任务时间线是什么?”）作为检索触发条件，人工标注应当被召回的记忆 ID，然后自动化测试该记忆是否被召回。  
**依据**：根据常见的记忆系统评测方法（如 Mem0 memory-benchmarks 的单轮检索）设计，无直接引用文献。  
**V1/V1.1 归属**：V1 必做。  
**对 V3 的具体改动**：实现检索流程的可测化调用接口，输出检索结果供评测使用；在测试中记录 `memory_used` 等埋点数据，计算准确率和污染率。  

### 2.1.5 关键指标  
**结论**：应监控并量化以下关键指标：**检索准确率（Accuracy/Recall@K）**：定义为正确相关的记忆被召回的比例（例如与人工标注相关记忆出现在 top_k 中的比例）。目标一般应达到高水平（参考 Mem0 在 LoCoMo/LongMemEval 上分别 ~92.5%、94.4%；可设 >=90%）。**覆盖率（Coverage）**：系统记忆库中包含项目实际关键信息的比率，可通过事后对比已发生成果来衡量。**检索延迟（Latency）**：端到端检索响应时间，包括向 Mem0 索引查询和结果返回，理想应＜0.5-1 秒。例如向量检索通常在 100–200ms 级。**增长率（Growth Rate）**：记忆数据库条目数增长速度，可用“每天新增记忆数”衡量，以评估系统容量扩展需求。**一致性失效率**：当记忆系统包含过期或冲突信息时，模型输出错误的比率，可使用类似 FAMA（Forgetting-Aware Memory Accuracy）这样的指标来量化（FAMA 惩罚使用过时信息）。**降级/失败率**：检索失败或降级触发的比例，例如 Mem0 调用失败后 SQLite 备用模式启动的情况。目标设置上：确保**检索准确率**＞90%；**检索延迟**多数情况下＜500ms（可参考上下文类型进行公允值）；**降级率**极低（＜5%）且每次降级都应成功返回正确记忆。其他可埋点指标包括 memory_used、used_memory_ids、memory_backend 等日志字段，用于计算上述指标。  

**依据**：Mem0 性能报告和相关研究给出了类似的指标框架（Mem0 在 LoCoMo/LongMemEval 上的 recall 分别 ~92.5%、94.4%）；人机认知和硬件研究指出长上下文会导致检索准确性下降，可据此设置合理回退策略。同时，Stanford 等提出“FAMA”指标来惩罚过时记忆的使用。Elastic RAG 最佳实践强调时间和元数据过滤可提高回答一致性。  
**V1/V1.1 归属**：指标定义为 V1 必需；持续监控和调整目标值为后续迭代任务（V1.1 及以后）。  
**对 V3 的具体改动**：在 AgentEvent 中记录更多指标（如检索延迟、fallback 触发），在测试中校验是否达标；在 Mem0 配置或检索策略中加入时间/元数据过滤以减少一致性错误。  

### 2.2.6 top_k=12 → 注入6 的合理性  
**结论**：业界的检索-注入比通常较高：初始召回常取数十到上百条文档，再精选几条注入提示。Mem0 在官方评测中即采用了 top_k=200 的召回预算。将召回限制为 12 条可能过小，易漏掉相关记忆；但若召回过多则上下文噪声增多并增加计算开销。当前 12→6 的设置相当于 50% 的筛选比例，显得偏低。建议在 V1 阶段测试增加召回数（例如 50 或 100 条），再使用现有场景重排规则保留最重要的 6 条。这样可提高关键记忆的命中率，同时通过后续 rerank/过滤控制噪声。实际可通过实验验证不同 top_k 的召回准确率变化。  

**依据**：Mem0 的 BEAM/LoCoMo 实验设置为 top_k=200；许多 RAG 系统也先召回几十条再筛选。大量研究表明，提高召回数一般能提高召回率，但要用 reranker 或阈值避免噪声。因此建议尝试比当前更高的 top_k。  
**V1/V1.1 归属**：V1 就该优化（可直接调整实现）；若资源充裕，V1.1 精细调参。  
**对 V3 的具体改动**：可将 top_k 从 12 调大（如 50、100），并观察返回结果质量；必要时同时加强后续排序或过滤步骤。  

### 2.2.7 Reranker  
**结论**：重排序（reranking）方法包括：交叉编码器 (cross-encoder)，利用双向注意力机制直接对齐查询与每个候选记忆（精度高但计算昂贵）；LLM 作为 reranker，可对候选集打分或排序（更灵活但成本更高）；MMR (Maximal Marginal Relevance) 则用于增强检索结果的多样性，抑制语义重复。Cohen’s d 等统计量一般用于评估差异显著性，不常直接用于记忆检索排序。V1 阶段不使用 reranker 会使召回结果质量依赖于初始检索信号，可能导致精度损失（Mem0 单纯检索在 LoCoMo/LongMemEval 上已有 ~92–94% 的召回率，表明已有一定能力）。增加 reranker 可进一步提高前 K 项的相关性。V1.1 最低限度的实现可采用轻量级方法：比如用小型 cross-encoder 对前 10–20 条记忆重新打分，或让 LLM 在生成回答前以“从这些候选中挑最相关的”方式进行过滤。ConvMemory 等研究表明，小型模型也能接近大 cross-encoder 的效果。  

**依据**：ConvMemory 报告指出强交叉编码器准确但慢，小型模型蒸馏后也可实现类似召回质量并显著降低延迟。MMR 在检索中常用于减少冗余，提高多样性。  
**V1/V1.1 归属**：V1 阶段可先不做（已是V1设计）；V1.1 阶段考虑添加（如实现基础 reranker）。  
**对 V3 的具体改动**：后续版本可在 top_k 召回结果上增加 rerank 步骤，例如交叉编码器或 LLM 判断，以进一步提升召回质量。  

### 2.2.8 Mem0 多路召回与重排序的必要性  
**结论**：Mem0 本身就集成了多路召回（BM25、向量检索、实体匹配等）并融合成最终得分（即一个“无监督打分”）。这个复合得分往往已经具有较高质量——例如 Mem0 在标准基准上的召回 F1≈0.94。因此目前阶段不急需额外的 reranker。V1 可直接使用 Mem0 的组合得分进行排序。若后续发现某些查询仍然错误率高，再考虑加上显式 reranker。  

**依据**：Mem0 在 LoCoMo/LongMemEval 上的综合得分接近 92–94%，说明已有多源融合策略效果不错。  
**V1/V1.1 归属**：V1 就该直接用（无需改）；可在 V1.1 再评估是否额外添加 reranker。  
**对 V3 的具体改动**：继续使用 Mem0 返回的融合分数。若后续拓展，考虑在 Mem0 外部增加后处理。  

### 2.2.9 场景重排得分函数  
**结论**：目前设计中使用了固定的类型优先级加若干硬编码规则，这种方法简单直接但缺乏灵活性。业界通常会尝试学习排序模型或利用上下文信息对结果打分。例如，可对记忆-查询对训练轻量级排序模型，或用 LLM 在 prompt 中内敛评判候选顺序（如让模型自己选择最相关记忆）。也可以利用上下文信号（如对话状态、时间戳）来动态调整权重。具体建议：V1 继续使用现有的类型优先级规则以保证可控和可解释；在 V1.1 中，可尝试基于任务反馈数据训练一个简单的排序器，或在提示中引入“按照与当前项目目标相关性”这样的自然语言指导来微调排序；甚至可以探索 LLM-based rerank（比如将候选记忆文本拼给 LLM，让其选最相关的）。这些方法都能避免硬编码的局限性，使排序更符合实际语境。  

**依据**：现代检索系统通常结合会话历史和元信息进行上下文感知排序（例如 Wu et al. 等研究展示了基于会话上下文的排序策略）。严格来说暂无学术资料具体针对项目场景排行，但经验上可参考上下文检索原则。  
**V1/V1.1 归属**：V1 阶段保持硬编码优先级（以简单实现优先）；V1.1 及以后应考虑替换为学习或模型驱动的排序方案。  
**对 V3 的具体改动**：初期无改动；后续可增设学习排序模型或提示策略，以自动调整记忆优先级。  

### 2.3.10 Token 预算分配  
**结论**：上下文窗口为有限资源，需把代币分配给最关键的信息来源。建议依据优先级分配：例如对话历史（最近消息 + Workspace状态）和关键任务信息占较大比例（合计可达60%以上），记忆项和提案、时间线等占剩余比例。具体例子：在 32K 上下文中，可给 ProjectFlow Workspace 状态+最近消息留 ~15K 词，记忆(6 条)约 2–3K 词，计划/提案 ~5K 词，时间轴 ~5K 词，留下缓冲。Anthropic 建议只保留“最小必要的高信号信息”。同时注意长上下文性能衰减：研究发现模型在超长上下文（如 32K+）时，准确率会显著下降。因此无需将上下文塞满，应为不同部分留足头尾展示空间，避免“遗忘中间”问题。总之，优先保证与当前任务最相关的信息（项目目标、当前问题），剩余预算用于历史记忆和时间线。  

**依据**：Anthropic 的实践指出应在上下文中只保留必要的高信号信息；研究表明 LLM 在上下文接近极限时质量会陡降。  
**V1/V1.1 归属**：作为基础设计原则应从 V1 开始即应用。  
**对 V3 的具体改动**：根据实际上下文长度（如 8K、32K 等）重新评估各部分长度，必要时压缩记忆或对话内容（例如摘要），保证总 token 数在模型有效区间内。  

### 2.3.11 记忆条数 vs 内容长度  
**结论**：建议按 token 总量而非固定条数控制记忆注入。因为单条记忆长度不一，严格按条数限会导致信息过多或过少。可以设定所有注入记忆总长度阈值（例如 500–1000 词）并动态调整条数。ProjectFlow 的每条记忆（内容+理由）通常较简洁，若按平均估算约几十词；6 条记忆合计也许在 300–600 词量级。若一条异常长，优先截断或省略细节，以防占用太多上下文资源。  
此外，大窗口注意力机制易遗忘中间内容，故总 token 数不宜过大；将关键记忆保持在上下文末尾更有利（例如先注入其他上下文，再加上标记的记忆块），以确保模型关注。  

**依据**：研究指出模型在长上下文中对处于中部的信息关注度降低；Anthropic 提倡只保留高信号内容。  
**V1/V1.1 归属**：V1 即宜按 token 预算控制记忆量。  
**对 V3 的具体改动**：将注入策略改为“最多 N 词”而非固定 6 条。对过长记忆做摘要，必要时增加下采样。  

### 2.3.12 记忆污染防护  
**结论**：为避免无关记忆干扰推理，应对检索结果做过滤与强化相关性检查。具体做法：**设置相关性阈值**：仅保留语义相似度或 BM25 分数高于阈值的记忆，低于阈值则丢弃，以减少“语义漂移”带来的干扰。**启用时效和元数据过滤**：利用 `valid_until` 字段执行时间范围过滤，只检索近期有效的记忆；使用项目 ID、记忆类型等元信息进行过滤或加权（如同 Elastic 推荐中的元数据加权），确保返回与当前项目上下文匹配的记忆。**混合检索策略**：结合 BM25（精确匹配）和向量检索，或使用 RRF 融合结果，能降低纯语义检索带来的错召。也可在推理前让 LLM 用 chain-of-thought 判断检索到的记忆是否真正相关，从而进一步筛选。  

**依据**：《Context poisoning in LLMs》最佳实践指出，向量检索容易检索到语义相近但上下文不符的文档，导致模型答非所问；建议使用时间过滤和元数据优先级来排除过时或不相关文档。混合检索（RRF 等）可以同时利用关键词精确匹配和语义匹配，降低语义噪声。  
**V1/V1.1 归属**：上述过滤与阈值逻辑应自 V1 开始实施。  
**对 V3 的具体改动**：在检索查询中增加 `valid_until >= now` 条件或类似范围查询；配置 Mem0 过滤器或在应用层通过检索分数（如相似度/Cohen’s d）丢弃低质量结果。  

### 2.3.13 去重策略  
**结论**：针对同一决策产生多条记忆的情况，最好在写入或检索阶段去重或合并：**写入去重**：可借鉴 Mem0 的做法，在写入时进行查重（通过事件 ID、文本哈希或实体链接）。如果新的记忆与已有内容重复，则拒绝新写入或更新已有条目。**检索阶段合并**：检索到相似内容时，应用合并策略——例如如果两条记忆仅职责表述不同，可合并为一条摘要；或者只保留最新/最具体的条目。行业常用记忆图谱将相关记忆连结起来（Mem0 内部即用实体图链接事实），以消除冗余。建议对 ProjectFlow，若方向和边界记忆高度重叠，可只注入其中一条（按最匹配的类型）。  

**依据**：Mem0 文档提到在写入时进行上下文查重和哈希 dedupe，将相关实体链接起来；多数人工智能记忆架构（如上下文记忆研究）建议在记忆存储时合并相同事件信息。  
**V1/V1.1 归属**：去重应从 V1 写入路径即考虑，以减少冗余记忆存储。  
**对 V3 的具体改动**：改进写入逻辑：写入时查找相同 `scope` 或有关联的条目，合并或丢弃重复内容；检索结果中避免同一事件的多条等效记忆同时出现。  

### 2.4.14 时间衰减  
**结论**：建议对旧记忆进行显式衰减或淘汰。一种做法是在检索评分中引入时间衰减函数（如指数衰减或高斯衰减），使新近记忆权重更高。如 Elasticsearch Agent 实现对记忆时间戳应用高斯衰减因子。ProjectFlow 已用 `valid_until` 和 `superseded` 字段来淘汰过期记忆，这是基本的强制失效；但时间衰减可让记忆权重逐步降低，减少过时信息影响。研究提出了“FAMA”指标来惩罚使用过时记忆，这也支持倾向权重较新记忆。V1 实现建议：至少确保检索时过滤掉 `valid_until` 过期的条目；若要进一步优化，可结合记忆写入时间或最后更新来动态调整排序分数（例如一天前记忆乘以 0.9 衰减系数）。  

**依据**：Elastic 记忆 Agent 中对每条记忆使用高斯时间衰减函数，以动态降低旧事实的重要性；最新研究提出的 FAMA 指标专门惩罚对已过时信息的依赖。  
**V1/V1.1 归属**：时间过滤（如使用 `valid_until`）为 V1 必做；高级衰减策略可留到 V1.1。  
**对 V3 的具体改动**：如果技术上可行，可在检索查询中加入时间衰减权重（如按`create_time`字段应用 ES 衰减查询），并确保`superseded`标记的记忆不再被召回。  

### 2.4.15 “为什么当初这么做”检索时机  
**结论**：记忆检索触发时机可以灵活设计：一种方式是**每次执行流程前（每轮决策）**都固定注入当前上下文相关记忆；另一种方式是只有在特定技能调用时才主动检索（“agentic retrieval”）。例如 Elasticsearch Agent 让模型在每个对话轮调用 `recall_memory` 工具，这样模型可根据需要自行提取记忆。对于 ProjectFlow，基本的方案是在每个任务推理阶段前统一检索并注入相关记忆，确保所有决策都能参考历史信息。这符合 V3 当前的设计。进一步，V1.1 可考虑让 LLM 在需要深挖背景（如回答“为什么做此决策”）时再调用专门的记忆检索工具，这样可能节省部分不必要的注入。总体而言，鉴于项目推进每步都可能涉及过去决策，固定注入较为简单直接；但允许模型自主决策调用检索也是可行的高级方案。  

**依据**：Elasticsearch Agent 的实现中，每轮会话开始时模型都会调用记忆检索工具；也有研究建议让 LLM 在生成中途按需调用检索。无现成文献具体比较哪种更适用项目场景。  
**V1/V1.1 归属**：V1 即可采用“固定注入每轮”方案（符合当前设计）；V1.1 可以探索“模型驱动调用”作为优化。  
**对 V3 的具体改动**：如引入 agentic 模式，在需要时通过工具调用记忆检索；否则默认按现有流程在每次对话输入时统一插入记忆。  

### 2.5.16 V1 验收标准  
**结论**：当前 V3 描述的 12 条验收标准（如记忆注入格式、记录字段等）多为设计要求，但需要补充可量化、可测试的性能标准。例如：**检索准确度下限**（如核心场景问答至少召回率 > 90%）、**检索延迟上限**（例如检索操作在 200–500ms 内完成）、**检索覆盖率**（已知重要事件被记忆的比例）、**降级触发成功率**（Mem0 失效时 SQLite 回退能否始终成功供给合理结果），以及**一致性错误率**（模型输出中因过时/错误记忆导致的回答错误比率）。此外，应明确定义记忆使用的埋点字段含义、测试方法、及验收阈值。综上，建议在验收列表中增加：检索召回率、误召率（无关记忆占比）、典型检索延迟、Mem0 vs 回退命中率、记忆库增长率等量化指标，并将其设置为可自动检测的阈值条件。  

**依据**：评测框架和工业实践通常包括性能和准确性指标（如前文所列）；当前标准缺少这些关键的可测量项。参考 Mem0 和 Elastic 文档可知写入/检索延迟和准确率指标的重要性。  
**V1/V1.1 归属**：这些指标应在 V1 定义并实现埋点监控。  
**对 V3 的具体改动**：补充测试脚本和监控系统，包括记录检索调用时延、召回/降级统计和记忆一致性检查等，并在验收文档中加入对应的合格标准。  

**V1 必做项清单**（检索/评估核心）：  
- **实现记忆检索监测**：对每次检索记录时间、召回的记忆 ID、关联场景等（`memory_used`/`used_memory_ids`）。  
- **定义检索准确率评测**：根据预置场景问答集计算召回率和误召率（参考 LoCoMo/LongMemEval 评测方法）。  
- **过滤与衰减**：应用时间过滤（`valid_until`）和相似度阈值，确保注入记忆的相关性；考虑简易的指数/高斯衰减。  
- **重复去除**：写入时查重、检索时避免同一事件多条结果。  
- **Token 分配策略**：制定上下文中各部分（记忆、对话、提案、时间线）的大致长度分配。  
- **性能目标**：设置并验证检索延迟（<500ms）和召回率门槛（>=90%）。  
- **基础评测 Harness**：搭建简单项目示例，验证检索结果正确性（mock LLM 问答）。  

**V1.1+ 候选清单**：  
- **高级 reranker**：增添交叉编码器或小型学习排序模型提升检索精度。  
- **动态上下文排序**：用学习方法或 LLM prompt 调整记忆条目优先级，替换硬编码规则。  
- **Agentic 检索模式**：让模型按需调用记忆检索工具，而非每轮固定注入。  
- **完整 BEAM/LongMemEval 测评**：在 Mem0 平台上运行大规模基准以评估系统极限。  
- **Memory 图谱**：构建记忆图谱来跟踪事件关联和冲突，增强时序和冲突解析能力。  

**最小评测 harness 设计**：  
- **场景构造**：建立一个小型项目环境（包含几项任务、角色、里程碑），并手动插入对应 ProjectMemory 记录，例如“里程碑达成X原因”、“成员A被指派任务Y”等。  
- **任务列表**：列出若干检索场景（如澄清决策原因、查询资源分配、回顾风险处理等），为每个场景预设期望检索结果（相关记忆 ID）。  
- **执行流程**：利用固定的“假 LLM”逻辑或自动化脚本模拟调用检索接口：向 Mem0 搜索（top_k 返回）并应用 V3 的过滤/排序逻辑，将结果放入 `<project_memories>`。  
- **断言**：检查注入列表是否包含所有期待的记忆（召回准确率），并且不包含标记为“无关”的记忆（污染率）。例如，若问题要求“为什么要这么做”，则验证包含相应的“原因”记忆；若没有无关任务的记忆出现。  
- **指标**：计算召回率（匹配的相关记忆数/总相关记忆数）、精确率（注入的相关记忆数/总注入条数）和检索时间。目标是召回率接近100%，污染率接近0。可采用持续集成测试自动执行此 harness，并在每次系统更新后验证通过性。
