# T23.D — 风险 + 重规划 + 导出流测试

## 你的任务

验证 Agent 是否识别真实风险（有证据、有建议），重规划是否最小化调整，导出是否有价值，以及 replan → push → checkin → risk 循环是否闭环。

---

## 环境准备

### 1. 启动后端

```bash
cd backend
.venv\Scripts\Activate.ps1
python -m uvicorn app.main:app --reload --port 8000
```

### 2. 启动前端

新开一个终端：

```bash
cd frontend
npm install   # 首次需要
npm run dev
```

### 3. 加载种子数据（推荐完整版）

在浏览器访问 `http://localhost:8000/docs`，找到 `POST /api/seed/demo`，点击 "Try it out" → "Execute"。

完整种子数据包含：
- 6 个用户 + 1 个项目（active）
- 4 个阶段 + 10 个任务
- 3 个已 finalized 分工
- 1 个签到周期 + 3 个签到响应（其中小张报告 blocker：SQLite 外键约束报错）
- 3 个风险：dependency（中）、workload（高）、deadline（中）
- 5 个行动卡
- 5 个 Agent 事件（clarify/plan/breakdown/assign/push）

这个数据集已有风险和签到数据，可以直接测试风险分析和重规划。

### 4. 确认 LLM 配置

- **第一轮（mock 模式）**：`LLM_PROVIDER=mock`
- **第二轮（真实 LLM）**：`LLM_PROVIDER=openai`，`LLM_API_KEY=你的key`，`LLM_MODEL=gpt-4o-mini`

### 5. 关于多成员操作

导航栏右侧有**用户切换下拉菜单**，可以切换当前身份为工作区内任意成员。本轨道使用种子数据，数据中已包含多成员的签到响应，大部分操作不需要切换身份。如需补充签到数据，可切换到对应成员身份后在签到表单中提交。

- [ ] 风险类型为以下之一：deadline / dependency / workload / scope / review / assignment / checkin
- [ ] **真实 LLM 模式**：Agent 能识别至少 3 类不同风险（PRD 6.5 要求至少 3 类）

### D3 — 查看风险卡详情

1. 查看单个风险卡

## 横向评估清单

### Agent 输出质量

- [ ] 风险 evidence 是否来自实际数据（签到/任务状态/截止日期/依赖/负载/评审压力）
- [ ] 风险 severity 是否与 evidence 匹配（高危有充分证据）
- [ ] 风险 recommendation 是否可操作（有具体行动建议）
- [ ] Agent 能识别至少 3 类风险（deadline / dependency / workload / scope / review / assignment / checkin）
- [ ] 重规划调整是否最小化（不是推翻重来）
- [ ] 重规划 before/after 是否清晰
- [ ] 重规划 impact 分析是否合理
- [ ] 重规划 reason 是否有依据
- [ ] 重规划输出包含 stage_adjustments + task_changes + action_cards 三部分

### 高风险确认

- [ ] high severity 风险是否强制人工确认
- [ ] 确认前是否无法自动执行
- [ ] 确认流程是否清晰

### 交互体验

- [ ] 风险卡是否一目了然（类型、严重度、证据）
- [ ] 风险筛选是否方便
- [ ] 重计划 diff 是否易读（颜色区分新增/移除/调整）
- [ ] 时间线是否可追溯
- [ ] 导出操作是否简单

### 导出价值

- [ ] 评审摘要包含关键信息
- [ ] 格式可用（Markdown 格式正确）
- [ ] 不遗漏重要数据
- [ ] 无 None/null 值

### 循环闭环

- [ ] replan → push → checkin → risk → replan 循环顺畅
- [ ] 每轮循环数据累积
- [ ] 不出现重复或冲突数据
- [ ] 时间线完整记录

### Fallback 安全

- [ ] 风险分析 fallback 有合法默认值
- [ ] 重规划 fallback 不破坏已有数据
- [ ] Fallback 不包含 None 值
- [ ] Fallback 事件在时间线中标记为 fallback

### Agent 状态透明度

- [ ] 风险分析成功时，显示绿色成功提示
- [ ] 风险分析 fallback 时，提示明确标注"已使用基础建议"
- [ ] 风险分析失败时，显示红色错误 + 重试入口
- [ ] 重规划同样遵循 success/repaired/fallback/failed 区分
- [ ] fallback 风险/重规划被标注为基础建议，不伪装成 AI 成功
- [ ] 时间线中每条事件显示正确的状态（success / repaired / fallback / failed）
