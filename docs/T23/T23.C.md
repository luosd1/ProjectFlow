# T23.C — 执行推进 + 签到流测试

## 你的任务

验证 Agent 是否主动推进项目（生成行动卡），签到后 Agent 是否正确分析进度、更新任务状态、识别风险。

---

## 环境准备

### 1. 启动后端

```bash
cd backend
.venv\Scripts\Activate.ps1
python -m uvicorn app.main:app --reload --port 8000
```

### 2. 启动前端

新开一个终端：

```bash
cd frontend
npm install   # 首次需要
npm run dev
```

### 3. 加载种子数据（推荐完整版）

在浏览器访问 `http://localhost:8000/docs`，找到 `POST /api/seed/demo`，点击 "Try it out" → "Execute"。

完整种子数据包含：
- 5 个用户（Lin/Mia/Chen/Noor/Jay）
- 1 个项目 "AI Study Planner"（active，已有方向卡）
- 2 个阶段（Prototype Loop: active）
- 5 个任务（2 个 P0 in_progress + 2 个 P1 not_started + 1 个 P2 not_started）
- 3 个已 finalized 的分工提案
- 1 个签到周期 + 1 个签到响应（Mia 报告 blocker）
- 1 个风险（deadline 类型，high severity）
- 2 个行动卡（team_next_step + personal_task）

这个数据集已经处于执行阶段，可以直接测试推进和签到。

### 4. 确认 LLM 配置

- **第一轮（mock 模式）**：`LLM_PROVIDER=mock`
- **第二轮（真实 LLM）**：`LLM_PROVIDER=openai`，`LLM_API_KEY=你的key`，`LLM_MODEL=gpt-4o-mini`

### 5. 关于多成员操作

导航栏右侧有**用户切换下拉菜单**，可以切换当前身份为工作区内任意成员。

- 其他成员提交签到：先在导航栏切换到对应成员身份，再在签到表单中提交
- 运行主动推进 / 签到分析：切回项目创建者身份操作

- [x] 行动卡类型为以下之一：personal_task / team_next_step / reminder / risk_action / kickoff_tip / checkin_prompt / assignment_request
- [x] 行动卡包含 title、content、reason
- [x] 行动卡包含 goal（任务目标）
- [x] 行动卡包含 start_suggestion（启动建议）

### C6 — 创建签到周期

1. 点击 Tabs 区域的"签到与状态"标签
2. 找到签到表单

**注意：** 签到周期可能需要通过 API 创建。如果前端没有创建周期的入口，在 `http://localhost:8000/docs` 中找到 `POST /api/checkin-cycles`，使用以下参数：

```json

  "stage_id": "active阶段的ID",
  "cadence_days": 7,
  "start_date": "2026-05-31",
  "created_by_user_id": "你的用户ID"
}
```

**验证：**

- [ ] 签到周期创建成功
- [ ] 周期状态为 active

### C7 — 提交签到响应

1. 在签到表单中填写：
   - **关联任务**：选择一个分配给你的任务（下拉选择）
   - **完成了什么**：如"完成了数据库 Schema 设计，正在写 API 接口"
   - **有阻塞吗**：如"SQLite 外键约束报错，排查了 2 小时未解决"（故意报告 blocker）
   - **下一周期可用时间**：如 8
   - **信心水平**：选择"低"

2. 点击"提交签到"

**验证：**
- [x] 提交成功（绿色提示"签到提交成功"）
- [x] 表单清空
- [x] 可以再次提交（不同成员的签到）

**截图：** 签到表单

### C8 — 多成员签到

1. 在导航栏用户切换下拉中，切换到其他成员身份
2. 在签到表单中提交该成员的签到
3. 重复切换到不同成员，至少 2-3 个成员提交签到，其中：
   - 1 人报告 blocker + 低信心
   - 1 人正常进度 + 高信心
   - 1 人无 blocker + 中信心

通过 API 提交签到（`POST /api/checkin-cycles/{cycle_id}/responses`）：

```json
{
  "project_id": "项目ID",
  "stage_id": "阶段ID",
  "user_id": "其他成员ID",
  "task_id": "任务ID",
  "what_done": "完成了前端页面布局",
  "blocker": null,
  "available_hours_next_cycle": 12,
  "mood_or_confidence": "high"
}
```

**验证：**

- [x] 每个成员都能成功提交
- [x] 签到响应关联到正确的周期

## 横向评估清单

### Agent 输出质量

- [ ] 行动卡是否指向最高优先级任务
- [ ] 行动卡 reason 是否有依据
- [ ] 签到分析是否基于实际签到数据（而非编造）
- [ ] Agent 是否识别了 blocker
- [ ] 任务状态更新是否有依据
- [ ] 生成的风险是否有 evidence

### 推进感知

- [ ] 运行"主动推进"后，是否感觉到 Agent 在"推你一把"
- [ ] 行动卡是否可操作（有明确的下一步），而非纯信息
- [ ] 推荐行动卡高亮是否引起注意

### 签到分析深度

- [ ] Agent 是否识别了低信心成员
- [ ] Agent 是否识别了 blocker
- [ ] Agent 是否基于签到数据更新了任务状态
- [ ] Agent 是否生成了对应风险
- [ ] reason 是否引用了具体签到内容

### 交互体验

- [ ] 签到表单是否简洁（不过多字段）
- [ ] 行动卡操作是否一键完成
- [ ] 签到历史是否可查看
- [ ] 任务状态更新是否方便

### 数据一致性

- [ ] 签到响应关联正确的周期和成员
- [ ] 任务状态更新与签到分析一致
- [ ] 行动卡关联正确的任务和成员
- [ ] 刷新页面后数据一致
- [ ] 签到分析不会自动修改任务 owner——Check-in 只收集状态和更新任务状态，不改变分工

### Fallback 安全

- [ ] Agent 分析失败时，fallback 行动卡有合法默认值
- [ ] Fallback 签到分析不包含 None 值
- [ ] Fallback 不破坏已有数据

### Agent 状态透明度

- [ ] 主动推进成功时，显示绿色成功提示
- [ ] 主动推进 fallback 时，提示明确标注"已使用基础建议"
- [ ] 主动推进失败时，显示红色错误 + 重试入口
- [ ] 签到分析同样遵循 success/repaired/fallback/failed 区分
- [ ] fallback 行动卡被标注为基础建议，不伪装成 AI 成功
