## 结论

D 轨当前最大的阻塞不是风险卡本身，而是 `risk -> replan -> confirm/reject -> push -> checkin -> risk -> replan` 这条闭环没有接完整。

主要问题有四类：

- 重规划 Agent 输出没有进入前端可确认的 proposal 链路，导致 D7-D10、D15、D17 断掉。
- Agent 时间线没有显示 `success / repaired / fallback / failed` 状态，fallback 也容易被当成普通成功。
- 高危风险的人工确认语义没有落到 UI 和状态流里。
- 风险 evidence、导出内容、空数据 fallback 的展示和验收证据不足。

## P0-01 重规划链路没有接上

【对应步骤】D7-D10、D15、D17

【现象】点击“调整计划”后，后端 `/agent/replan` 可以返回 AgentFlowResult，但前端“风险与调整”里的“计划调整”面板一直显示空状态。当前 `ProjectDashboard` 固定传入：

```tsx
<ReplanDiff before={[]} after={[]} proposal={null} />
```

后端已有 `/api/replans/confirm` 和 `confirm_replan()`，并且 `backend/app/tests/test_checkin_risk_replan_flow.py` 能证明确认后可以修改阶段、任务并创建行动卡。但 `agent_flow_service.py` 对 `ReplanOutput` 没有持久化分支，前端也没有获取、确认、拒绝 replan proposal 的入口。

【期望】

- 点击“调整计划”后，`风险与调整` 标签下出现最新的重规划 proposal。
- 面板显示 `before`、`after`、`impact`、`reason`、`requires_confirmation`。
- 面板显示 `stage_adjustments`、`task_changes`、`action_cards`，并遵守最多 1 个阶段调整、1 个任务变更、1 个行动卡的最小化原则。
- 有“确认调整”和“拒绝调整”按钮。
- 确认后调用 `/api/replans/confirm`，阶段、任务、行动卡、仪表盘指标刷新。
- 拒绝后不持久化任何阶段、任务、行动卡变更，页面不崩溃。
- 不允许绕过 finalized assignment owner guard。

【建议修复点】

- `backend/app/services/agent_flow_service.py`：为 `ReplanOutput` 增加持久化路径，或创建专门的 ReplanProposal 存储。不要让 replan output 只存在于 AgentFlowResult 的一次性响应里。
- `backend/app/schemas/agent_flow.py`：如果继续走 proposal 模式，返回 `proposal_id`，并保证前端能根据项目拉取 pending replan。
- `frontend/src/lib/types.ts`：补充 replan proposal 类型。
- `frontend/src/lib/api.ts`：补充 confirm/reject replan API。
- `frontend/src/components/risk/replan-diff.tsx`：展示完整 proposal，并提供确认/拒绝操作。
- `frontend/src/components/project/project-dashboard.tsx`：不要再传 `proposal={null}`，改为真实 latest pending replan。
- `frontend/src/app/projects/[projectId]/page.tsx`：接上 confirm/reject handler，成功后刷新项目状态。

【验收】

- D7：点击“调整计划”后出现重规划结果。
- D8：能看到 before/after、impact、reason 和 diff 数量 badge。
- D9：确认后任务看板、阶段看板、行动卡同步更新。
- D10：拒绝后不持久化变更，页面不崩溃。
- D15：replan 后可以继续 run push，循环可继续。

## P0-02 Agent 时间线没有显示真实 status

【对应步骤】D11、Fallback 安全、Agent 状态透明度

【现象】后端 `AgentEventRead` 已返回 `status`，但前端 `AgentEvent` 类型没有 `status` 字段，`AgentTimeline` 也没有渲染 `success / repaired / fallback / failed`。反馈中“时间线中每条事件显示正确的状态”这一条是误判，实际没有显示。

【期望】

- 时间线每条事件都显示状态 badge：`success`、`repaired`、`fallback`、`failed`。
- fallback/failed 有明确视觉区分。
- fallback 文案不能伪装成普通 AI 成功，应显示“已使用基础建议”或等价中文。
- Agent 操作完成提示要根据 AgentFlowResult.status 区分：
  - `success`：绿色成功。
  - `repaired`：提示“已修复格式后完成”。
  - `fallback`：提示“已使用基础建议”。
  - `failed`：红色错误并保留重试入口。

【建议修复点】

- `frontend/src/lib/types.ts`：为 `AgentEvent` 增加 `status: "success" | "repaired" | "fallback" | "failed"`。
- `frontend/src/components/agent/timeline.tsx`：渲染 status badge，并按状态着色。
- `frontend/src/app/projects/[projectId]/page.tsx`：`runAgent()` 需要读取 `AgentFlowResult`，不要只要请求成功就显示“已完成”。
- `frontend/src/components/project/project-dashboard.tsx`：成功/降级/失败提示文案分开。

【验收】

- D11 时间线每条事件显示 event_type、status、reasoning_summary、created_at。
- fallback 事件在时间线中清楚标记为 fallback。
- fallback 风险/重规划不被显示成普通成功。

## P0-03 高危风险确认流程不清晰

【对应步骤】D14、横向评估“高风险确认”

【现象】`RiskAnalysisOutput` schema 会校验 high severity 时 `requires_confirmation=true`，但当前风险分析结果会直接持久化为风险卡。前端没有把 `requires_confirmation` 展示出来，也没有说明“确认前不会执行什么”。用户只能看到风险卡的接受、忽略、解决，这不足以覆盖 T23.D 的高风险确认流程。

【期望】

- high severity 风险有红色高危 badge。
- 高危风险的建议不能自动修改任务、阶段、owner 或行动卡。
- UI 明确区分“接受风险记录”和“执行建议”。
- 如果高危建议需要触发重规划或行动卡，应先要求用户确认。
- 确认前不会自动执行任何计划调整。

【建议修复点】

- `frontend/src/components/risk/risk-card.tsx`：高危卡片增加确认语义说明，比如“高危建议需确认后执行”。
- `frontend/src/components/risk/risk-panel.tsx`：高危计数和筛选保持可见。
- `backend/app/services/agent_flow_service.py`：确认 risk analysis 不会因为 high severity 自动创建 action card 或修改任务、阶段。
- 如需要把 high-risk recommendation 转成动作，走 replan proposal 或 action confirmation，不要直接落库为执行结果。

【验收】

- D14 能清楚看到 high severity 的特殊标识。
- 确认前没有任务、阶段、owner、行动卡被自动修改。
- 用户能理解下一步应该“接受风险”还是“确认调整计划”。

## P0-04 accepted 风险无法 resolved

【对应步骤】D6

【现象】风险卡按钮只在 `risk.status === "open"` 时渲染。点击“接受”后状态变为 accepted，按钮消失，无法继续点击“解决”。这和 D6“选择一个已接受的风险，点击解决”的验收项冲突。

【期望】

- open 风险可以接受、忽略、解决。
- accepted 风险仍可以解决。
- ignored 和 resolved 风险不需要继续显示操作按钮。
- 风险数量统计在状态变化后同步刷新。

【建议修复点】

- `frontend/src/components/risk/risk-card.tsx`：accepted 状态下显示“解决”按钮。
- `frontend/src/components/risk/risk-panel.tsx`：筛选计数刷新后正确显示 accepted/resolved。
- `backend/app/services/risk_service.py`：如需要，增加状态迁移约束，但至少允许 accepted -> resolved。

【验收】

- D4 接受后，风险进入“已接受”筛选。
- D6 在“已接受”中能点击解决，状态变为 resolved，并进入“已解决”筛选。

## P1-01 风险分析返回数量和“至少 3 类风险”要求冲突

【对应步骤】D1、Agent 输出质量

【现象】T23.D 要求真实 LLM 模式下 Agent 能识别至少 3 类风险，但当前 risk prompt 明确要求“Return exactly 1 highest-risk item”。这会让真实 LLM 很难满足 D1 和横向评估。

【期望】

- 如果 T23.D 保持“至少 3 类风险”，风险分析应允许一次返回最多 3 个有证据的风险，类型尽量覆盖 `deadline / dependency / workload / scope / review / assignment / checkin` 中的不同类别。
- 每个风险必须有实际 evidence，不要为了凑 3 类编造。
- 没有足够证据时可以少于 3 个，但真实 LLM + demo seed/blocker 场景应能稳定产出至少 3 类。

【建议修复点】

- `backend/app/agent/prompts.py`：去掉 `Return exactly 1 highest-risk item`，改为“return up to 3 concrete risks with different types when evidence exists”。
- `backend/app/agent/modules/risk_analysis.py`：同步 user_prompt。
- `backend/app/agent/output_schemas.py`：确认 `RiskAnalysisOutput.risks` 支持多项，并补测试。
- `backend/app/services/risk_service.py`：去重逻辑不要误伤不同类型风险。
- `backend/app/seed/demo_projectflow.py`：demo seed 已有 dependency/workload/deadline 三类风险，可以作为验收基线。

【验收】

- 真实 LLM 模式下，demo seed/blocker 场景能识别至少 3 类风险。
- 每条风险都有来自任务、签到、截止日期、依赖、负载或评审压力的 evidence。
- severity 和 evidence 匹配。

## P1-02 evidence 展示含 raw JSON、变量名或英文 key

【对应步骤】D3、Agent 输出质量

【现象】后端 `RiskRead.evidence` 支持 `list[str | dict]`，但前端 `normalizeRisk()` 会把非字符串 evidence `JSON.stringify()` 成字符串。这样 `RiskCard` 里处理 dict 的逻辑基本不会触发，用户可能看到 raw JSON、英文 key、变量名或 ID。

【期望】

- evidence 展示为可读中文。
- 结构化 evidence 可显示来源和事实，比如“来源：签到；事实：小张报告 SQLite 外键约束报错”。
- 不直接展示 raw JSON、`task_id`、`stage_id`、`available_hours_next_cycle` 这类内部字段名，除非同时有中文解释。

【建议修复点】

- `frontend/src/lib/types.ts`：把 `Risk.evidence` 改为 `Array<string | Record<string, unknown>>`。
- `frontend/src/lib/api.ts`：`normalizeRisk()` 不要把 dict 全部 stringify；应保留 dict 或转成中文文本。
- `frontend/src/components/risk/risk-card.tsx`：为常见 key 做中文映射，如 `source -> 来源`、`detail/text -> 事实`、`task_title -> 任务`、`member_name -> 成员`。
- `backend/app/agent/prompts.py`：约束用户可见文本用中文，避免输出只有内部字段的 evidence。

【验收】

- D3 风险卡 evidence 不出现 raw JSON。
- evidence 能说明来源和具体事实。
- 真实 LLM 模式下 evidence 不编造、不只显示 ID。

## 原反馈中需要修正的判断

以下内容不要原样照抄到 issue 或 PR 里：

- “未实现重规划”：不准确。后端 confirm API 存在，问题是 Agent replan output 没有进入前端可展示、可确认、可拒绝的 proposal 链路。
- “时间线中每条事件显示正确的状态”：错误。当前前端没有显示 status。
- “D1 推荐改英文”：表述不清。实际应检查用户可见文本是否统一中文；如果看到英文，应改中文。
- “未 fallback 纵使确实不成功”：表述不清。应改为“fallback 结果没有在 UI 和时间线中明确标记，用户无法区分基础建议和正常 AI 成功”。
- “D15 无效”：应拆成可定位问题，核心是重规划链路断裂导致循环无法开始。
